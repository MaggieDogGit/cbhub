Implement a full Financial Market Infrastructure (FMI) taxonomy model in CBHub.

Objective
Create a scalable FMI taxonomy that:
1. Covers payment FMIs and non-payment FMIs
2. Separates schemes, clearing mechanisms, settlement systems, and networks
3. Keeps CLS separate
4. Keeps card schemes separate
5. Supports securities FMIs such as CREST
6. Can be extended later with operating hours, currencies, reachability, one-leg-out support, and participation logic

Important modelling principles
1. Do not mix payment schemes and processing infrastructures in the same subtype
2. Do not classify CLS under payment systems; keep it as separate FX settlement infrastructure
3. Do not classify card schemes together with account-to-account payment schemes
4. Support explicit relationships between:
   - scheme -> clearing mechanism
   - clearing mechanism -> settlement system
5. Support FMI types outside payments, including:
   - CSD
   - SSS
   - CCP
   - Trade Repository
6. The taxonomy must be globally usable, not SEPA-only

Required example mappings
Ensure the taxonomy can correctly classify the following examples:

Payment schemes
- SEPA Credit Transfer -> Payment Systems / Payment Schemes / Credit Transfer Scheme
- SEPA Instant Credit Transfer (SCT Inst / OCT Inst) -> Payment Systems / Payment Schemes / Instant Credit Transfer Scheme
- SEPA Direct Debit -> Payment Systems / Payment Schemes / Direct Debit Scheme

Payment clearing mechanisms
- STEP2 SCT -> Payment Systems / Payment Clearing Mechanisms / ACH Credit Transfer Clearing
- RT1 -> Payment Systems / Payment Clearing Mechanisms / ACH Instant Credit Transfer Clearing
- STEP2 SDD -> Payment Systems / Payment Clearing Mechanisms / ACH Direct Debit Clearing
- EURO1 -> Payment Systems / Payment Clearing Mechanisms / High-Value Net Clearing

Payment settlement systems
- T2 -> Payment Systems / Payment Settlement Systems / RTGS
- TIPS -> Payment Systems / Payment Settlement Systems / Instant Settlement System
- Fedwire Funds -> Payment Systems / Payment Settlement Systems / RTGS

Payment connectivity / messaging
- SWIFT -> Payment Systems / Payment Connectivity / Messaging Networks / Financial Messaging Network
- Nexus-style interoperability arrangements -> Payment Systems / Payment Connectivity / Messaging Networks / Payment Interoperability Network

FX settlement infrastructure
- CLS -> FX Settlement Infrastructure / FX Payment-versus-Payment Settlement System

Card payment infrastructure
- Visa -> Card Payment Infrastructure / Global Card Scheme
- Mastercard -> Card Payment Infrastructure / Global Card Scheme
- domestic debit card schemes should be supported under Domestic Card Scheme

Securities market infrastructure
- CREST -> Securities Market Infrastructure / Securities Settlement System
- Euroclear Bank -> Securities Market Infrastructure / International Central Securities Depository
- Clearstream -> Securities Market Infrastructure / International Central Securities Depository
- local CSDs should be supported under Central Securities Depository

CCPs
- LCH -> Central Counterparties / Derivatives CCP or Multi-Asset CCP depending on implementation granularity
- Eurex Clearing -> Central Counterparties / Derivatives CCP or Multi-Asset CCP

Trade repositories
- DTCC trade repository entities -> Trade Repositories / Derivatives Trade Repository
- UnaVista -> Trade Repositories / Derivatives Trade Repository

Required relationship model
Add explicit support for FMI relationships. At minimum, support these relationship types:

1. SCHEME_USES_CLEARING_MECHANISM
   Example:
   - SCT uses STEP2 SCT
   - SCT Inst uses RT1
   - SCT Inst uses TIPS where relevant

2. CLEARING_MECHANISM_SETTLES_IN_SETTLEMENT_SYSTEM
   Example:
   - STEP2 SCT settles in T2
   - STEP2 SDD settles in T2
   - RT1 settles in TIPS
   - EURO1 settles in T2

3. FMI_OPERATED_BY_ORGANIZATION
   Example:
   - RT1 operated by EBA Clearing
   - EURO1 operated by EBA Clearing
   - T2 operated by Eurosystem / ECB
   - CREST operated by Euroclear UK & International
   - CLS operated by CLS Group / CLS Bank

4. CARD_SCHEME_HAS_PROCESSING_INFRASTRUCTURE
   Keep optional for now but model-ready

5. SECURITIES_INFRASTRUCTURE_LINKS_TO_PAYMENT_FMI
   Keep optional for now but model-ready, for future DvP analysis

Naming conventions
Use the following naming conventions consistently:

Top-level label
- Financial Market Infrastructures

Use singular subtype labels in the taxonomy tables if that is the app standard, but keep display names readable.
Examples:
- RTGS
- Instant Settlement System
- Credit Transfer Scheme
- ACH Credit Transfer Clearing
- Financial Messaging Network
- Central Securities Depository

Avoid overly local acronyms as taxonomy labels unless globally understood.
Example:
- prefer “ACH Instant Credit Transfer Clearing” over “CSM”
- store local acronyms as aliases, not as taxonomy labels

Validation rules
1. A Payment Scheme cannot be directly classified as a Payment Settlement System
2. CLS cannot be placed under Payment Systems
3. Visa and Mastercard cannot be placed under Payment Schemes for account-to-account payments
4. CREST cannot be placed under Payment Systems
5. EURO1 cannot be classified as ACH
6. T2 cannot be classified as a clearing mechanism
7. SCT Inst / OCT Inst must be classifiable as a scheme, not only as infrastructure

Migration / refactor guidance
1. Preserve existing FMI records where possible
2. Map current categories into the new taxonomy
3. If existing categories are too broad or incorrect, migrate them to the closest correct category
4. Keep aliases / legacy labels where useful for search
5. Do not break existing references unnecessarily
